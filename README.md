# K-Means Performance Benchmark: CPU vs. GPU

A comprehensive performance analysis of the K-Means clustering algorithm across different hardware architectures and frameworks. This project benchmarks five implementations — from a pure sequential CPU baseline to GPU-accelerated variants on both NVIDIA and AMD hardware.

## 🚀 Features

- **CPU Baseline (Python):** Sequential implementation using `scikit-learn` — correctness and timing reference.
- **CPU Baseline (C++):** Hand-written K-Means in pure C++17 with HPC-oriented flat memory layout — the true apples-to-apples baseline for GPU speedup measurement.
- **GPU High-Level (NVIDIA):** Accelerated implementation using NVIDIA `RAPIDS cuML`, interface identical to the CPU baseline.
- **AMD iGPU — Taichi (Vulkan):** K-Means on the Radeon 780M via [Taichi](https://taichi-lang.org/). Runs on the AMD iGPU using the Vulkan backend — `pip install taichi`, no ROCm or OpenCL drivers required.
- **Custom CUDA Kernel *(WIP)*:** Hand-written C++/CUDA kernel with explicit thread-level parallelism for NVIDIA GPUs.
- **Performance Profiling:** All variants save results to CSV and feed into a unified visualization script producing time comparison and speedup charts.

## ⚙️ HPC Design Decisions

Every implementation shares the same architectural choices to ensure **fair comparison**.

| Decision | Rationale |
|---|---|
| `float32` everywhere | GPUs are optimized for FP32; `double` would skew memory bandwidth comparisons |
| Flat 1D row-major array in C++ | No heap fragmentation — identical layout to GPU VRAM |
| Squared Euclidean distance (no `sqrt`) | Monotonic — doesn't change nearest-centroid winner, saves cycles |
| Isolated timing | Timer wraps strictly the Lloyd loop — data generation and I/O excluded |
| `n_init=1` in scikit-learn | Default of 10 runs would be unfair vs cuML and C++ which run once |
| `algorithm="lloyd"` in scikit-learn | Forces standard Lloyd, identical to all GPU implementations |
| K-Means++ with `random_state=42` | Deterministic init across all variants for reproducible convergence |
| Taichi `ti.atomic_add` | Native GPU atomic float — no CAS loop hack like OpenCL 1.2 |

## ▶️ Quick Start

### NVIDIA variants — Google Colab
1. Open `notebooks/KMeans_Benchmark.ipynb`
2. `Runtime > Change runtime type > GPU (T4)`
3. Run all cells in order

### CPU variants — local
```bash
git clone https://github.com/andreisugu/KMeans-GPU-Benchmark.git
cd KMeans-GPU-Benchmark
pip install -r requirements.txt

# Python — scikit-learn
cd src/baseline
python benchmark_cpu.py --skip-large    # Small + Medium
python benchmark_cpu.py                 # all 4 scenarios

# C++ sequential
make && ./kmeans_seq --all
```

### AMD iGPU — Taichi Vulkan (local, Radeon 780M)
```bash
# One-time setup — that's it
pip install taichi

# Verify AMD GPU is detected
python -c "import taichi as ti; ti.init(arch=ti.vulkan); print('Vulkan OK')"

# Run
cd src/taichi
python benchmark_taichi.py --skip-large --validate   # Small + Medium + inertia check
python benchmark_taichi.py                            # all 4 scenarios
```

### Visualize all results
```bash
cd results
python plot_results.py    # generates time_comparison.png + speedup_chart.png
```
